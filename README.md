#Задание 1

Активный master-сервер

   Все изменения (INSERT/UPDATE/DELETE) выполняются на master, поэтому схема простая и без конфликтов.
   Slave получает данные репликацией, его можно использовать для чтения отчётов и тем самым разгрузить master.
   Slave подходит как резерв, при сбое master можно переключиться на slave.

Master-несколько slave

   Масштабирование чтения, запросы SELECT распределяются по нескольким репликам.
   Выше отказоустойчивость по чтению, при падении одной реплики остаются другие.
   Можно разделить задачи, одна реплика под отчёты, другая под бэкапы и т.д.
